本文档从北京移动 AI 质检平台工程实践中提取，供其他项目复用。文档聚焦于可落地执行的工程约束，非项目特定的业务内容已抽象为通用描述。

## 2. 架构约束

### 2.1 子系统边界

若项目涉及多技术栈/多子系统，必须遵循以下原则：

1. **职责分离**：各子系统各有独立职责域，不允许互相替代核心能力。
2. **契约先行**：子系统间通过 HTTP REST 通信，接口契约必须版本化、可追溯，落成仓库文件而非口头约定。
3. **数据隔离**：允许共用同一数据库实例，但必须通过 schema ownership 隔离，禁止跨系统无边界共享表或互改对方 owned 数据。
4. **发布流程**：一方可生成配置/规则/模型等发布物，但生产执行必须通过受控发布流程进入主控链路，不得绕过。

### 2.2 前端组织方式

1. **单工程优先**：第一阶段采用单前端工程，通过路由域区分不同业务域（如 `/ops/**`、`/ai/**`）。
2. **模块隔离**：各业务模块不得互相直接依赖页面级实现，共享逻辑下沉到 `components/`、`services/`、`stores/`。
3. **预留拆分空间**：模块结构应支持未来平滑拆分为独立应用。

### 2.3 仓库目录结构

```
项目根/
├─ apps/                  # 可部署应用（前端 + 各后端服务）
├─ packages/              # 共享领域模型、契约、公共工具
├─ infra/                 # 部署编排、CI/CD、环境定义
│  └─ docker/
│     ├─ compose.dev.yml
│     ├─ compose.verify.yml
│     └─ env/             # 环境变量模板
├─ scripts/               # 仓库级自动化脚本
├─ docs/                  # 项目正式知识库
└─ [项目名]Docs/          # 业务原始需求资料
```

## 3. 本地开发环境

### 3.1 总体原则

1. **本地开发默认走 Docker Compose**，不以宿主机裸环境作为标准路径。
2. 必须区分 `dev`（日常开发）与 `verify`（机械验证）两类模式。
3. 所有服务的本地启动、构建、验证命令，必须优先提供 Docker 入口或其脚本包装。
4. 宿主机裸环境仅允许作为临时诊断手段，不得作为任务验收、提交前检查或 CI 对齐依据。

### 3.2 dev 模式要求

- 支持源码挂载
- 支持热更新或快速重启
- 日志可直接查看
- 允许按服务粒度单独启动

### 3.3 verify 模式要求

- 接近 CI 的运行方式
- 不依赖热更新
- 执行构建、类型检查、单测和基础 smoke
- 验证失败以明确退出码结束

### 3.4 每个服务必须补齐的材料

每新增一个正式服务，必须先补齐以下内容再进入正式开发：

1. `Dockerfile.dev`（dev 模式容器定义）
2. `Dockerfile.verify`（verify 模式容器定义）
3. Compose 服务定义（写入 `infra/docker/compose.dev.yml` / `compose.verify.yml`）
4. 对应脚本入口或明确的 Docker 命令
5. 至少一条可自动执行的 verify 命令

### 3.5 数据库本地接入

1. 本地数据库服务通过 Docker Compose 提供。
2. 使用命名卷保存开发数据（dev），临时容器用于验证（verify）。
3. 环境变量模板提供 `.example` 文件，真实凭据文件不提交到共享仓库。
4. 初始化脚本放在 `infra/docker/<db>/init/`，按文件名顺序执行。
5. 提供增量同步脚本，支持将新增初始化 SQL 同步到已有开发卷。

## 4. 容器化与部署

### 4.1 容器化要求

1. 所有服务必须容器化，生产环境运行于 Kubernetes。
2. 服务启动方式、健康检查、日志和配置管理按容器化思路设计。
3. 容器镜像版本固定，避免使用 `latest` tag。

### 4.2 Kubernetes 部署要求

1. 生产部署资源定义放在 `infra/` 目录。
2. 本地开发环境与生产环境分层说明，不可混写。
3. 必须包含：健康探针（readiness/liveness）、资源限制、滚动更新策略。
4. 配置文件通过 ConfigMap/Secret 注入，不得硬编码到镜像。

### 4.3 制品管理

1. 产出明确的制品封装和 provenance（来源证明）。
2. 生成 SPDX 格式的 SBOM（软件物料清单）。
3. 对 release artifact 生成签名 attestation。

## 5. CI/CD 要求

### 5.1 基础门禁（CI）

1. **仓库级检查**：文档骨架完整性、仓库卫生。
2. **代码质量**：lint、format:check、typecheck、build、test。
3. **安全扫描**：依赖审查、OSV 漏洞扫描、供应链安全评分。
4. **脚本检查**：Shell/PowerShell 脚本语法校验。

### 5.2 持续交付（CD）

1. 优先产出明确制品和 provenance，不过早假设部署目标。
2. 接入真实技术栈时，在现有流水线上扩展，不另起临时脚本。
3. CI/CD 中所有 Action/插件固定到 commit SHA，不允许使用浮动版本标签。

### 5.3 推荐接入顺序

1. 先建立基础 CI 门禁（文档 + 代码卫生）。
2. 逐步叠加项目特定验证命令。
3. 用真实构建产物替换占位打包逻辑。
4. 技术栈和环境稳定后补充部署 job。
5. 即使交付方式变化，SBOM 和 provenance 等供应链能力应保留。

## 6. 安全基线

### 6.1 凭据管理

1. 本地开发和验证环境仅允许使用环境变量或 env 文件提供凭据。
2. 应用配置不得内置真实密码，也不得通过代码级 fallback 伪造可用凭据。
3. 环境变量模板（`.example`）只保留示例值，真实文件不提交到共享仓库。
4. 外部 API 密钥、数据库密码、AI 模型密钥不得写入仓库或聊天记录。

### 6.2 数据安全

1. 多系统共用数据库时必须通过 schema ownership 隔离。
2. 共享数据库划分建议：业务 schema、AI/算法 schema、共享注册 schema、审计 schema。
3. 敏感数据（身份证号、人脸图片、签名文件）存储加密方式需专项确认。
4. 数据分级、脱敏与保留策略需明确。

### 6.3 接口安全

1. 内部 API 调用是否启用 mTLS 需明确。
2. 统一鉴权方向成立，SSO、Token 传递、服务间认证方案需专项确认。
3. 尚未接入统一认证的本地 smoke 端点不能按生产接口暴露。

### 6.4 供应链安全

1. 锁定并提交项目真实依赖的 lockfile。
2. 在 PR 上做依赖变更审查，阻止高风险依赖引入。
3. 用 OSV 扫描已知漏洞。
4. 用 OpenSSF Scorecard 做仓库级安全姿态分析。
5. 构建过程尽量可重复、可验证。

## 7. 测试与质量门禁

### 7.1 测试分层

1. **开发阶段**：lint、format:check、typecheck、build、test 全部通过。
2. **验证阶段**：Docker verify 模式执行机械验证。
3. **上线前补齐**：
   - 业务单元测试
   - 接口契约测试
   - 端到端 smoke
   - 验收报告模板

### 7.2 质量门禁清单

每次实质性变更后必经检查：

| 检查项 | 方式 | 通过标准 |
|--------|------|----------|
| 类型检查 | `npm run typecheck`（前端）/ IDE 类型检查（后端） | 零错误 |
| 构建 | `npm run build` / `./mvnw package` / Docker build | 构建成功 |
| 代码规范 | ESLint / Prettier / Checkstyle | 零警告 |
| 单测 | Vitest / JUnit / pytest | 全部通过 |
| 安全扫描 | 依赖审查 / OSV | 无高危漏洞 |
| 文档完整性 | 文档骨架检查 | 必需文档齐全 |

### 7.3 验收标准

1. 无验收标准的任务不得自动标记完成。
2. 验收报告模板应包含：检查项、通过标准、实际结果、差异说明。
3. 所有结果可追溯到：需求来源 -> 任务计划 -> 执行记录 -> 验收报告。

## 8. 代码规范

### 8.1 通用原则

1. **KISS（简单至上）**：优先选择简单、清晰、可观测的方案，拒绝不必要的复杂性。
2. **YAGNI（精益求精）**：仅实现当前明确所需的功能，不过度设计。
3. **DRY（杜绝重复）**：识别重复代码模式，主动抽象和复用。
4. **SOLID**：单一职责、可扩展接口、子类型可替换、接口专一、依赖抽象。

### 8.2 提交规范

1. Commit 遵循 Conventional Commits：`<type>(<scope>): <subject>`。
2. Commit 范围清晰，不提交无关文件。
3. 提交前确认文档、示例、脚本是否反映最终状态。
4. 复杂或高风险改动先写执行计划（execution plan）。

### 8.3 代码审查四维检查

每次 MR/PR 必做：

1. **逻辑正确性**：边界条件、空指针、并发安全、业务规则符合性。
2. **规范符合**：命名、注释覆盖率、包结构一致性。
3. **性能风险**：N+1 查询、大对象内存、同步阻塞。
4. **安全隐患**：SQL 注入、XSS、权限校验缺失、硬编码敏感信息。

## 9. 文档纪律

### 9.1 必需文档

| 文档 | 用途 | 要求 |
|------|------|------|
| AGENTS.md | AI Agent 导航入口 | 只做路由，不堆规则 |
| ARCHITECTURE.md | 架构总览（权威来源） | 覆盖目标结构、边界、技术方向 |
| LOCAL_DEVELOPMENT.md | 本地开发说明 | 环境搭建、命令入口、故障排除 |
| SECURITY.md | 安全约束 | 凭据、认证、数据分级 |
| CICD.md | CI/CD 说明 | 流水线结构、接入方式 |
| REPO_COLLAB_GUIDE.md | 协作约定 | 提交、文档同步、测试约定 |
| CONTRIBUTING.md | PR 前检查项 | 操作清单 |
| FRONTEND.md | 前端约定（如涉及） | 技术栈、目录结构、验收标准 |

### 9.2 文档维护原则

1. 行为一旦变化，对应文档必须在同一次改动里同步更新。
2. 文档之间引用保持一致，不允许矛盾或过期的内容。
3. 推荐新增边界清楚的小文档，而非继续往大文档堆内容。
4. 架构变更是 `ARCHITECTURE.md` 的唯一变更来源。
5. 技术选型变更必须写 ADR（架构决策记录）。

### 9.3 ADR（架构决策记录）

重大技术决策必须记录为 ADR，包含：
- 状态（Proposed / Accepted / Deprecated / Superseded）
- 日期
- 决策人
- 背景（为什么需要决策）
- 候选方案及优劣
- 决策结论及原因
- 影响范围和后续动作

## 10. AI 辅助开发约束

如项目采用 AI Agent 辅助开发，必须遵守：

### 10.1 文件边界控制

1. 自动化任务必须明确 `include_files`（允许修改）和 `exclude_files`（禁止触碰）。
2. 自动化执行不得突破文件边界。
3. 涉及部署、密钥、权限、安全配置的任务默认要求人工确认。

### 10.2 执行可追溯

所有自动化结果可追溯到：
```
需求来源 -> 任务计划(plan.md) -> 任务契约(task contract) -> 执行记录 -> 验收报告
```

### 10.3 环境一致性

1. AI 执行器进行本地运行、构建或验证时，默认必须使用 Docker 命令。
2. 不得把宿主机裸环境验证结果作为默认结论。
3. task plan 中的验证命令应逐步替换为 Docker 入口或其脚本包装。

## 11. 环境启动清单

### 11.1 开发环境首次搭建

```bash
# 1. 复制环境变量模板
cp infra/docker/env/*.env.example infra/docker/env/*.env

# 2. 按需编辑 env 文件（密码、端口等）

# 3. 启动开发环境
docker compose -f infra/docker/compose.dev.yml up -d --build
```

### 11.2 验证流程

```bash
# 全部服务验证
docker compose -f infra/docker/compose.verify.yml up --build --abort-on-container-exit
```

### 11.3 各服务端口约定（示例）

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端 | 5173 | Vite dev server |
| 业务后端 | 8080 | Spring Boot |
| AI 后端 | 8000 | FastAPI/Uvicorn |
| 数据库 | 5432 | PostgreSQL |

## 12. 上线前必须补齐项

在进入生产环境前，以下内容为强制要求：

1. **测试基线**：业务单元测试、接口测试、端到端 smoke、验收报告模板。
2. **部署方案**：部署拓扑图、上线回滚方案、制品版本策略、配置发布策略。
3. **安全**：统一鉴权、服务间认证、权限模型、审计追踪、密钥管理方案。
4. **可观测性**：日志字段规范、健康检查、timeout/retry/backoff 策略、指标采集、链路追踪。
5. **发布物协议**：规则/流程/配置发布物的数据结构和版本策略。
6. **数据策略**：数据分级、脱敏、保留策略、备份恢复方案。

## 附录：关键术语速查

| 术语 | 含义 |
|------|------|
| dev 模式 | Docker Compose 本地开发模式，支持热更新和源码挂载 |
| verify 模式 | Docker Compose 机械验证模式，接近 CI，一次性的构建/类型检查/单测/smoke |
| schema ownership | 数据库 schema 级所有权隔离，禁止跨系统无边界互写 |
| SBOM | 软件物料清单，记录所有依赖组件及版本 |
| provenance | 制品来源证明，签名认证的 build provenance attestation |
| ADR | 架构决策记录，记录重大技术决策的背景、方案和影响 |
| include/exclude | AI 自动化执行的文件边界控制 |
